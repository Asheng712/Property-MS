## 第一章 项目概述

**负责人**：钱富安、王林峰

### 1.1 项目背景

#### 行业痛点

- 传统物业管理依赖人工催缴、纸质工单，效率低、易出错
- 业主缴费需到物业中心排队，报修/投诉反馈周期长
- 物业公司缺乏数字化手段进行财务核算与工单调度
- 社区公告与通知无法精准触达业主

#### 项目目标

构建前后端分离的智慧物业管理系统，覆盖 **PC 管理端**与**移动端**，实现资产→合同→账单→支付→核销全链路数字化，提供自动化费用计算、在线支付、一键核销等核心能力，支持报修调度、投诉处理、公告发布等社区服务场景。

#### 目标用户

| 角色 | 终端 | 核心场景 |
|:----|:----|:-------|
| 物业管理员 | PC 管理端 | 资产监管（楼栋/单元/房间树形管理）、财务核算（账单生成+支付核销+报表导出）、工单调度（报修派发+投诉处理+状态跟踪）、运营看板（实时统计待收费/欠费/投诉/报修） |
| 社区住户 | 移动端 | 账单查询与在线缴费（多账单合并支付）、一键报修（名下房屋选择+进度跟踪）、投诉建议提交与处理结果查看、资产购买/租赁申请、社区公告阅读 |

双端共享同一套后端 API，通过 Nginx 统一代理。

### 3.1 完整目录树

```
Property-MS/
├── backend/
│   ├── pom.xml                        # 父 POM, Spring Boot 3.2.12 parent

│   ├── wisdom-common/                 # Result, PageResult, BaseContext, 枚举, AI接口
│   │   └── src/main/java/com/wisdom/
│   │       ├── result/                # Result<T> (code+msg+data), PageResult<T> (total+records)
│   │       ├── context/               # BaseContext (ThreadLocal)
│   │       ├── enumeration/           # BillStatus, FeeType, PayMethod, PaymentStatus
│   │       └── ai/                    # AIService, RepairAnalysisResult
│   ├── wisdom-pojo/                   # 14 个 Entity (严禁业务逻辑)
│   │   └── src/main/java/com/wisdom/entity/
│   │       ├── User, Role
│   │       ├── House, Contract, PurchaseApplication
│   │       ├── Bill, BillBatch, PaymentRecord, PaymentRecordBill, PropertyFeeConfig
│   │       └── Repair, Complaint, Notice, FileTask
│   └── wisdom-server/                 # 核心业务模块
│       ├── src/main/java/com/wisdom/
│       │   ├── Application.java       # @SpringBootApplication + @MapperScan
│       │   ├── config/                # MybatisPlusConfig, SecurityConfig (BCrypt+CORS), WebMvcConfig (空)
│       │   ├── controller/            # 11 个 Controller
│       │   ├── service/               # 9 个接口 + impl/ 下 9 个实现
│       │   ├── mapper/                # 14 个 Mapper
│       │   ├── dto/                   # 29 个 DTO
│       │   ├── vo/                    # 18 个 VO
│       │   ├── filter/                # JwtAuthenticationFilter (44行) + AuthenticationFilter (132行)
│       │   ├── handler/               # GlobalExceptionHandler
│       │   ├── exception/             # BusinessException
│       │   ├── util/                  # JwtTokenUtil, SecurityUtil
│       │   ├── annotation/            # @LoginRequired (被Filter使用), @Anonymous (未被使用)
│       │   └── ai/impl/               # AIServiceImpl (Stub)
│       └── src/main/resources/
│           ├── application.yml
```

#### 代码解析：双 Filter 认证链

**JwtAuthenticationFilter**（`filter/JwtAuthenticationFilter.java`，44 行）— 第一道 Filter（`@Order(1)`），只负责提取，不负责阻断：

```java
@Component
@Order(1)
@Slf4j
public class JwtAuthenticationFilter implements Filter {

    @Autowired
    private JwtTokenUtil jwtTokenUtil;

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        String token = httpRequest.getHeader("Authorization");

        if (token != null && token.startsWith("Bearer ")) {
            token = token.substring(7);
            if (jwtTokenUtil.validateToken(token)) {
                Long userId = jwtTokenUtil.getUserIdFromToken(token);
                if (userId != null) {
                    BaseContext.setCurrentId(userId);
                }
            }
        }

        try {
            chain.doFilter(request, response);
        } finally {
            BaseContext.removeCurrentId();
        }
    }
}
```

JwtAuthenticationFilter 只做"尽力而为"的 Token 提取，即使 Token 无效也不阻断请求。这允许后续 AuthenticationFilter 根据不同路径的 `@LoginRequired` 标注来决定是否需要鉴权。

**AuthenticationFilter**（`filter/AuthenticationFilter.java`，132 行）— 第二道 Filter（`@Order(2)`），负责鉴权决策和 401 阻断：

```java
@Component
@Order(2)
@Slf4j
public class AuthenticationFilter implements Filter {

    @Autowired
    private JwtTokenUtil jwtTokenUtil;

    @Autowired
    private RequestMappingHandlerMapping handlerMapping;

    @Autowired
    private ObjectMapper objectMapper;

    private static final String[] WHITE_LIST = {
            "/api/v1/auth/login",
            "/api/v1/auth/register",
            "/swagger-ui/",
            "/swagger-ui.html",
            "/api-docs/",
            "/v3/api-docs/",
            "/knife4j/"
    };

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        HttpServletResponse httpResponse = (HttpServletResponse) response;
        String requestUri = httpRequest.getRequestURI();

        if (isWhiteList(requestUri)) {
            chain.doFilter(request, response);
            return;
        }

        if (!requiresLogin(httpRequest)) {
            chain.doFilter(request, response);
            return;
        }

        String token = httpRequest.getHeader("Authorization");
        if (token == null || !token.startsWith("Bearer ")) {
            sendErrorResponse(httpResponse, 401, "UNAUTHORIZED");
            return;
        }

        token = token.substring(7);
        if (!jwtTokenUtil.validateToken(token)) {
            log.warn("Invalid token for URI: {}", requestUri);
            sendErrorResponse(httpResponse, 401, "INVALID_TOKEN");
            return;
        }

        Long userId = jwtTokenUtil.getUserIdFromToken(token);
        if (userId == null) {
            sendErrorResponse(httpResponse, 401, "UNAUTHORIZED");
            return;
        }
        BaseContext.setCurrentId(userId);

        try {
            chain.doFilter(request, response);
        } finally {
            BaseContext.removeCurrentId();
        }
    }

    private boolean isWhiteList(String uri) {
        for (String path : WHITE_LIST) {
            if (uri.startsWith(path)) {
                return true;
            }
        }
        return false;
    }

    private boolean requiresLogin(HttpServletRequest request) {
        try {
            HandlerExecutionChain handlerChain = handlerMapping.getHandler(request);
            if (handlerChain == null) {
                return false;
            }
            if (handlerChain.getHandler() instanceof HandlerMethod handlerMethod) {
                Method method = handlerMethod.getMethod();
                return method.isAnnotationPresent(LoginRequired.class);
            }
            return false;
        } catch (Exception e) {
            log.debug("Handler mapping failed for: {}", request.getRequestURI(), e);
            return false;
        }
    }

    private void sendErrorResponse(HttpServletResponse response, int status, String message)
            throws IOException {
        response.setStatus(status);
        response.setContentType("application/json;charset=UTF-8");
        Result<Void> result = Result.error(message);
        response.getWriter().write(objectMapper.writeValueAsString(result));
    }
}
```

使用 `RequestMappingHandlerMapping.getHandler()` 动态获取目标 HandlerMethod，再反射检查 `@LoginRequired` 注解。这种方式比传统的路径前缀匹配更精确——Controller 中标注了 `@LoginRequired` 的方法才需要鉴权，未标注的接口（即使不在白名单）也不校验 Token。

**两 Filter 的分工**：

|                | JwtAuthenticationFilter   | AuthenticationFilter |
|:-------------- |:------------------------- |:-------------------- |
| 职责             | Token 提取 + 写入 ThreadLocal | 鉴权决策 + 401 阻断        |
| @Order         | 1（先执行）                    | 2（后执行）               |
| Token 无效时      | 静默跳过                      | 阻断返回 401             |
| ThreadLocal 清理 | ✓ finally 块               | ✓ finally 块          |

### 4.4 统一 API 响应

**Result<T>**（`wisdom-common/result/Result.java`）——项目中最核心的基础设施类，所有 Controller 返回值必须包装在此：

```java
    private String msg;
    private T data;
```

**GlobalExceptionHandler**（`handler/GlobalExceptionHandler.java`）——异常 → HTTP 响应映射，从具体到宽泛逐级捕获。注意 `RuntimeException` 和 `Exception` 是两个独立的 `@ExceptionHandler` 方法，日志级别不同：

```java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<Result<Void>> businessExceptionHandler(BusinessException ex) {
        log.warn("Business exception: code={}, message={}", ex.getCode(), ex.getMessage());

        HttpStatus status = switch (ex.getCode()) {
            case 401 -> HttpStatus.UNAUTHORIZED;
            case 403 -> HttpStatus.FORBIDDEN;
            case 404 -> HttpStatus.NOT_FOUND;
            case 400 -> HttpStatus.BAD_REQUEST;
            default  -> HttpStatus.OK;
        };

        Result<Void> body = Result.error(ex.getMessage());
        body.setCode(ex.getCode());
        return ResponseEntity.status(status).body(body);
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Result<Void>> validationExceptionHandler(
            MethodArgumentNotValidException ex) {
        String message = ex.getBindingResult().getFieldErrors().stream()
                .map(error -> error.getField() + ": " + error.getDefaultMessage())
                .collect(Collectors.joining(", "));
        log.warn("Validation exception: {}", message);
        return ResponseEntity.badRequest().body(Result.error(message));
    }

    @ExceptionHandler(HttpMessageNotReadableException.class)
    public ResponseEntity<Result<Void>> httpMessageNotReadableExceptionHandler(
            HttpMessageNotReadableException ex) {
        log.warn("Bad request payload: {}", ex.getMessage());
        return ResponseEntity.badRequest().body(Result.error("请求体格式错误"));
    }

    @ExceptionHandler(NoResourceFoundException.class)
    public ResponseEntity<Result<Void>> noResourceFoundExceptionHandler(
            NoResourceFoundException ex) {
        return ResponseEntity.notFound().build();
    }

    @ExceptionHandler(RuntimeException.class)
    public ResponseEntity<Result<Void>> runtimeExceptionHandler(RuntimeException ex) {
        log.error("Runtime exception: {}", ex.getMessage(), ex);
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
                .body(Result.error("服务器内部错误"));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<Result<Void>> exceptionHandler(Exception ex) {
        log.error("System exception: {}", ex.getMessage(), ex);
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
                .body(Result.error("服务器内部错误"));
    }
}
```

Service 层 `throw BusinessException.badRequest("...")` → GlobalExceptionHandler 拦截 → 根据 code 映射 HTTP 状态码 → 返回 `{"code":400,"msg":"...","data":null}`。前端 `http.ts` 检测 `code !== 200` 统一 toast 提示。

**BaseContext**（`wisdom-common/context/BaseContext.java`）——基于 ThreadLocal 的请求上下文，是贯穿双 Filter → Service 的用户身份载体：

```java
public class BaseContext {
    public static ThreadLocal<Long> threadLocal = new ThreadLocal<>();

    public static void setCurrentId(Long id)    { threadLocal.set(id); }
    public static Long getCurrentId()           { return threadLocal.get(); }
```

Tomcat 每个请求由独立线程处理，ThreadLocal 天然实现线程隔离，无需在方法间显式传递 `userId` 参数。`removeCurrentId()` 调用在 Filter 的 `finally` 块中，确保线程归还线程池时不会泄露上一个请求的用户信息。

### 4.6 关键业务：缴费闭环


#### 代码解析：缴费闭环核心实现

**账单生成**（`FinanceServiceImpl.doGenerateBills()`）——按费用类型 + 月份批量为名下房产生成账单：

```java
private void doGenerateBills(BillGenerateDTO dto) {
    Integer feeType = dto.getFeeType();
    String billMonth = dto.getBillMonth() != null
        ? dto.getBillMonth()
        : LocalDate.now().format(DateTimeFormatter.ofPattern("yyyy-MM"));

    if (FeeTypeEnum.PROPERTY.getCode().equals(feeType)) {
        billMonth = LocalDate.now().plusMonths(1)
            .format(DateTimeFormatter.ofPattern("yyyy-MM"));
    }

    PropertyFeeConfig config = null;
    if (FeeTypeEnum.PROPERTY.getCode().equals(feeType)) {
        config = propertyFeeConfigMapper.getEffectiveConfig(billMonth);
        if (config == null) {
            throw BusinessException.badRequest(
                "未找到" + billMonth + "的物业费配置，请先设置单价");
        }
    }

    List<House> houses;
    if (dto.getHouseId() != null) {
        House house = assetMapper.selectById(dto.getHouseId());
        if (house == null) throw BusinessException.notFound("房屋不存在");
        houses = List.of(house);
    } else {
        houses = assetMapper.selectAllAssets();
    }

    int count = 0;
    for (House house : houses) {
        if ("BUILDING".equals(house.getType())) continue;
        if (house.getOwnerId() == null) continue;

        LambdaQueryWrapper<Bill> dupQuery = new LambdaQueryWrapper<>();
        dupQuery.eq(Bill::getHouseId, house.getId())
                .eq(Bill::getFeeType, feeType)
                .eq(Bill::getBillMonth, billMonth)
                .ne(Bill::getStatus, BillStatusEnum.VOIDED.getCode());
        if (billMapper.selectCount(dupQuery) > 0) continue;

        List<Contract> contracts = contractMapper.selectByHouseId(house.getId());
        if (contracts.isEmpty() && !FeeTypeEnum.PROPERTY.getCode().equals(feeType))
            continue;

        Bill bill = new Bill();
        bill.setUserId(house.getOwnerId());
        bill.setHouseId(house.getId());
        bill.setFeeType(feeType);
        bill.setBillNo("BILL-" + feeType + "-"
            + LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyyMMddHHmmssSSS"))
            + "-" + count);
        bill.setBillMonth(billMonth);
        bill.setStatus(BillStatusEnum.PENDING.getCode());
        bill.setDueDate(YearMonth.parse(billMonth).atEndOfMonth());
        bill.setRemark(dto.getRemark() != null ? dto.getRemark() : "");
        bill.setCreateTime(LocalDateTime.now());

        BigDecimal amount = calculateBillAmount(feeType, house, contracts, config);
        bill.setAmount(amount);

        if (!contracts.isEmpty()) {
            bill.setContractId(contracts.get(0).getId());
        }

        billMapper.insert(bill);
        count++;
    }

    if (count == 0) {
        throw BusinessException.badRequest(
            "没有符合条件的房屋可生成账单（可能已有同期账单或无可计费房屋）");
    }
}
```

`doGenerateBills()` 在执行 INSERT 前先 `selectCount` 检查同房屋+同费用类型+同月份是否已有未作废账单，防止重复生成。`generateBillsSafe()` 包装此方法并吞掉异常，用于合同审批通过后的自动账单生成——即使自动生成失败也不会中断审批流程。

**用户提交缴费**（`FinanceServiceImpl.submitPayment()`）——多账单合并支付。实际代码有两个校验循环（第一个循环含一个逻辑瑕疵——`continue` 放过了待缴费状态的账单但没有累加金额，第二个循环才做正确的状态校验和金额累加）：

```java
@Transactional(rollbackFor = Exception.class)
public PaymentRecordVO submitPayment(PaymentSubmitDTO dto) {
    Long uid = userService.getRequiredCurrentUserId();
    List<Bill> bills = billMapper.selectBatchIds(dto.getBillIds());

    if (bills.isEmpty()) throw BusinessException.badRequest("未找到账单");
    if (bills.size() != dto.getBillIds().size())
        throw BusinessException.badRequest("部分账单不存在");

    BigDecimal totalExpected = BigDecimal.ZERO;
    for (Bill bill : bills) {
        if (!bill.getUserId().equals(uid))
            throw BusinessException.badRequest("账单不属于当前用户");
        if (BillStatusEnum.PENDING.getCode().equals(bill.getStatus()))
            continue;
        throw BusinessException.badRequest("账单 " + bill.getBillNo()
            + " 状态不正确（当前: "
            + BillStatusEnum.fromCode(bill.getStatus()).getLabel() + "）");
    }

    for (Bill bill : bills) {
        if (!bill.getUserId().equals(uid))
            throw BusinessException.badRequest("账单不属于当前用户");
        if (!BillStatusEnum.PENDING.getCode().equals(bill.getStatus())) {
            throw BusinessException.badRequest("账单 " + bill.getBillNo()
                + " 状态不正确（当前: "
                + (BillStatusEnum.fromCode(bill.getStatus()) != null
                    ? BillStatusEnum.fromCode(bill.getStatus()).getLabel()
                    : "未知") + "）");
        }
        totalExpected = totalExpected.add(bill.getAmount());
    }

    PaymentRecord pr = new PaymentRecord();
    pr.setPaymentNo("PAY-" + LocalDateTime.now()
        .format(DateTimeFormatter.ofPattern("yyyyMMddHHmmss")));
    pr.setUserId(uid);
    pr.setAmount(totalExpected);
    pr.setPayMethod(dto.getPayMethod());
    pr.setPayTime(LocalDateTime.now());
    pr.setStatus(PaymentStatusEnum.PENDING.getCode());
    pr.setVoucherUrl(dto.getVoucherUrl() != null ? dto.getVoucherUrl() : "");
    pr.setRemark(dto.getRemark() != null ? dto.getRemark() : "");
    pr.setCreateTime(LocalDateTime.now());
    paymentRecordMapper.insert(pr);

    for (Bill bill : bills) {
        PaymentRecordBill prb = new PaymentRecordBill();
        prb.setPaymentRecordId(pr.getId());
        prb.setBillId(bill.getId());
        prb.setAmount(bill.getAmount());
        prb.setCreateTime(LocalDateTime.now());
        paymentRecordBillMapper.insert(prb);

        bill.setStatus(BillStatusEnum.SUBMITTED.getCode());
        billMapper.updateById(bill);
    }

    return toPaymentRecordVO(pr);
}
```

**核销与撤销**——管理员操作的对称设计：

```java
@Transactional(rollbackFor = Exception.class)
public void verifyPayment(Long id) {
    PaymentRecord pr = paymentRecordMapper.selectById(id);
    if (pr == null) throw BusinessException.notFound("支付记录不存在");
    if (!PaymentStatusEnum.PENDING.getCode().equals(pr.getStatus())) {
        throw BusinessException.badRequest("仅待核销状态的记录可以核销");
    }

    Long adminId = BaseContext.getCurrentId();

    pr.setStatus(PaymentStatusEnum.VERIFIED.getCode());
    pr.setVerifyUser(adminId);
    pr.setVerifyTime(LocalDateTime.now());
    paymentRecordMapper.updateById(pr);

    LambdaQueryWrapper<PaymentRecordBill> prbQw = new LambdaQueryWrapper<>();
    prbQw.eq(PaymentRecordBill::getPaymentRecordId, id);
    List<PaymentRecordBill> details = paymentRecordBillMapper.selectList(prbQw);
    for (PaymentRecordBill d : details) {
        Bill bill = billMapper.selectById(d.getBillId());
        if (bill != null) {
            bill.setStatus(BillStatusEnum.PAID.getCode());
            billMapper.updateById(bill);
        }
    }
}

@Transactional(rollbackFor = Exception.class)
public void cancelPayment(PaymentCancelDTO dto) {
    PaymentRecord pr = paymentRecordMapper.selectById(dto.getId());
    if (pr == null) throw BusinessException.notFound("支付记录不存在");
    if (!PaymentStatusEnum.VERIFIED.getCode().equals(pr.getStatus())) {
        throw BusinessException.badRequest("仅已核销状态的记录可以撤销");
    }

    Long adminId = BaseContext.getCurrentId();

    pr.setStatus(PaymentStatusEnum.CANCELLED.getCode());
    pr.setCancelUser(adminId);
    pr.setCancelTime(LocalDateTime.now());
    pr.setCancelReason(dto.getCancelReason());
    paymentRecordMapper.updateById(pr);

    LambdaQueryWrapper<PaymentRecordBill> prbQw = new LambdaQueryWrapper<>();
    prbQw.eq(PaymentRecordBill::getPaymentRecordId, dto.getId());
    List<PaymentRecordBill> details = paymentRecordBillMapper.selectList(prbQw);
    for (PaymentRecordBill d : details) {
        Bill bill = billMapper.selectById(d.getBillId());
        if (bill != null && BillStatusEnum.PAID.getCode().equals(bill.getStatus())) {
            bill.setStatus(BillStatusEnum.PENDING.getCode());
            billMapper.updateById(bill);
        }
    }
}
```

每个操作方法入口都检查当前状态是否允许目标操作——`verifyPayment()` 只能在 PENDING 状态下执行，`cancelPayment()` 只能在 VERIFIED 状态下执行。状态不对直接 throw `BusinessException`，防止并发覆盖和非法状态转换。

**关键业务规则汇总**：

| # | 规则 | 实现位置 |
|:--|:-----|:-------|
| 1 | 费用类型固定四种（租金/买房金额/押金/物业费），枚举化管理 | `FeeTypeEnum.java` |
| 2 | 不允许部分缴费——用户支付金额必须等于所选账单合计 | `submitPayment()` 双循环校验 |
| 3 | 一个支付记录可对应多张账单（多账单合并支付） | `bus_payment_record_bill` 关联表 |
| 4 | 管理员核销后才更新账单为已缴费——用户提交后进入"待核销"中间状态 | `verifyPayment()` |
| 5 | 核销后允许撤销——撤销后账单回退为待缴费，操作记录完整保留 | `cancelPayment()` |
| 6 | 物业费调价从设置时的下一个月开始生效——历史账单金额不重新计算 | `PropertyFeeConfig.getEffectiveConfig()` |
| 7 | 账单生成前进行幂等检查——同房屋+同费用类型+同月份+非作废 → 跳过 | `doGenerateBills()` repeat detection |

**账单与支付记录双维度状态流转对照**：

```
账单状态 (BillStatusEnum):              支付记录状态 (PaymentStatusEnum):

待缴费 (0) ──用户提交──→ 待核销 (1)        待核销 (0)
   ↑                      │                  │
   │ 撤销回退              │ 管理员核销        │ 管理员核销
   │                      ↓                  ↓
   │                   已缴费 (2)          已核销 (1)
   │                      │                  │
   └──撤销─────────────── ← (可撤销)          │ 撤销
                                              ↓
已作废 (4) — 枚举已定义，无作废 API 端点    已撤销 (3)

                                        已驳回 (2) — 枚举已定义，但无 API 触发入口
```

`BillStatusEnum.VOIDED(4, "已作废")` 和 `PaymentStatusEnum.REJECTED(2, "已驳回")` 已在枚举中定义，但 FinanceController 中没有对应的作废/驳回 API 端点，当前无代码路径可达这两个状态。VOIDED 仅在 `doGenerateBills()` 的幂等检查中作为排除条件使用。

所有涉及账单状态和支付记录状态联动变更的操作，均通过 `@Transactional(rollbackFor = Exception.class)` 保证。三个核心事务操作：提交缴费（插入支付记录 + 插入支付明细 + 更新账单 0→1）、核销（更新支付记录 0→1 + 更新账单 1→2）、撤销（更新支付记录 1→3 + 更新账单 2→0），均同一事务。

#### 4.8.1 关键架构决策

##### 决策 1：为什么是 Vue 3 + Vite？

项目实际使用的技术栈（`frontend/package.json` / `frontend-mobile/package.json`）：

| 维度 | 选型 | 代码证据 |
|:----|:-----|:-------|
| 框架 | Vue 3.5 (PC) / 3.5 (移动) | `"vue": "^3.5.13"` → `^3.5.34"` |
| 构建 | Vite 6.2 (PC) / 8.0 (移动) | 原生 ESM 开发服务器，HMR 极速反馈 |
| UI 库 | Element Plus 2.14 (PC) / Vant 4.9 (移动) | 两个生态分别服务管理员和业主场景 |

##### 决策 2：为什么自研双 Filter 而非 Spring Security？

源码证据：`filter/JwtAuthenticationFilter.java`（44 行）+ `filter/AuthenticationFilter.java`（132 行）= **176 行**，commit `5eaa859`（"修复Spring Security编译失败及投诉/报修接口鉴权缺失"）移除了 `spring-boot-starter-security` 依赖。

| 维度 | Spring Security（移除前） | 自研双 Filter（当前） |
|:----|:----------------------|:-----------------|
| 依赖 | `spring-boot-starter-security`（全家桶） | 仅 `spring-security-crypto`（BCrypt） |
| 配置量 | SecurityFilterChain Bean + CSRF 禁用 + CORS 配置 + 白名单 | 两个 `@Component` + `@Order` 注解 |
| 白名单机制 | URL 前缀匹配，改白名单需改配置类重新编译 | `RequestMappingHandlerMapping.getHandler()` 动态反射，注解驱动 |
| Token 逻辑 | 需自定义 `OncePerRequestFilter` 子类 | JwtAuthenticationFilter 直接 `implements Filter` |
| 认证链 | AuthenticationManager → Provider → UserDetailsService | 无——本项目不需要角色层级/权限表达式 |

**核心权衡**：Spring Security 的认证链是为"多种认证方式 + 复杂权限模型"设计的。本项目只需要"验 JWT → 把 userId 放 ThreadLocal"，176 行 Java 覆盖全部鉴权需求，引入整个安全框架是过度设计。

##### 决策 3：为什么使用 MyBatis-Plus（而非 JPA / 原生 MyBatis）？

14 个 Mapper 全部为 **纯注解接口**（`com/wisdom/mapper/` 下 0 个 XML 文件），继承 `BaseMapper<T>` 即可获得 CRUD：

| 特性 | 实际用法 | 优势 |
|:----|:-------|:----|
| 查询构造 | `LambdaQueryWrapper<Bill>` + 方法引用 `.eq(Bill::getUserId, uid)` | 字段名编译期检查；编译为 `PreparedStatement` 防注入 |
| 动态筛选 | 链式追加 `qw.eq(...).like(...).orderByDesc(...)` | 多条件组合（费用类型+月份+状态+房屋）逐条追加，代码扁平可读 |
| 数据权限 | Service 层追加 `.eq(Bill::getUserId, uid)` | Lambda 表达式的动态条件追加自然契合"非 admin 自动过滤"需求 |

JPA 的 `Specification` / `CriteriaBuilder` 可以实现等价功能，但本项目 9 个 Service 实现类全部使用 LambdaQueryWrapper 模式——MyBatis-Plus 是项目的事实标准。

##### 决策 4：Maven 多模块的编译期隔离

```
wisdom-pm (父 POM, Spring Boot 3.2.12 parent)
├── wisdom-common    依赖: 纯 JDK + Lombok
│   └── Result<T>, PageResult<T>, BaseContext, 4 枚举, AIService 接口
├── wisdom-pojo      依赖: Lombok + MyBatis-Plus（仅注解层）
│   └── 14 个 @TableName Entity 类
└── wisdom-server    依赖: Spring Boot + common + pojo + 所有运行时依赖
    └── Controller / Service / Mapper / Filter / Handler / Config / DTO / VO
```

| 模块 | 隔离效果 | 代码证据 |
|:----|:-------|:-------|
| `wisdom-common` | 不含 Spring 依赖 → `@Service` / `@Component` 在编译期不可用 | `pom.xml` 仅依赖 lombok |
| `wisdom-pojo` | 不含 Spring → `@Autowired` 编译报错 → **物理上杜绝 Entity 膨胀为 DAO** | `pom.xml` 仅 lombok + `mybatis-plus-boot-starter`（注解层） |
| `wisdom-server` | Controller/Service/Mapper/Filter/DTO/VO 全在 server，无法跨模块引用 | 唯一含 `@SpringBootApplication` 的模块 |

##### 决策 5：PC 端 fetch vs 移动端 Axios

双端 HTTP 层核心共同点：都检测 `result.code !== 200` → 401/403 跳转登录 → 正常则解包返回 `result.data`。

| 维度 | PC 端 | 移动端 |
|:----|:-----|:------|
| 文件 | `services/http.ts`（114 行） | `services/http.ts`（51 行） |
| 库 | 原生 `fetch` | Axios 1.16.1 |
| Token 注入 | 请求前手写 `new Headers()` + `headers.set('Authorization', ...)` | Axios 请求拦截器自动 `config.headers.Authorization = ...` |
| 超时 | 无默认（浏览器默认 ~90s） | `timeout: 15000`（15 秒） |
| 参数清洗 | `buildUrl()` 过滤 `null/undefined/''` | 无（依赖调用方传干净参数） |
| 401 处理 | `CustomEvent('wisdompm:auth-expired')` → 解耦 Pinia | `window.location.hash = '#/login'`（Hash 路由无需服务端配合） |
| 请求监控 | `monitor.recordRequest()`（耗时+成功率） | 无 |

##### 决策 6：sessionStorage（而非 localStorage）存 Token

PC 端：`setToken()` → 先 `localStorage.removeItem(TOKEN_KEY)` 清旧值 → 再 `sessionStorage.setItem(TOKEN_KEY, token)`。移动端：直接 `sessionStorage.setItem(TOKEN_KEY, token)`。

| 维度 | sessionStorage | localStorage |
|:----|:-------------|:------------|
| 生命周期 | 关闭标签页 → 自动清除 | 持久化到手动删除 |
| XSS 风险 | 攻击者关闭标签页后 token 失效 | token 永久可读 |
| 多标签页 | 每个标签页独立登录 | 共享登录态 |

**选择原因**：物业管理系统涉及业主个人信息和缴费金额，**安全性 > 便利性**。关闭标签页=自动登出，无需手动操作。

##### 决策 7：移动端 Hash 路由 vs PC 端 History 路由

`frontend/src/router/index.ts` → `createWebHistory()` | `frontend-mobile/src/router/index.ts` → `createWebHashHistory()`

| 维度 | History (PC) | Hash (移动) |
|:----|:------------|:----------|
| URL 形态 | `/dashboard` | `/#/dashboard` |
| 服务端配置 | **必须** Nginx `try_files $uri /index.html` | **无需**——`#` 后面内容不发往服务端 |
| 部署约束 | 必须有 Nginx / Vercel rewrites | 任意静态文件服务器均可 |

PC 端部署环境固定（Nginx 已配置 `try_files`），用 History 模式 URL 更干净。移动端 Hash 模式使部署零服务端依赖。

##### 决策 8：为什么用 Docker Compose 而非 K8s？

`compose.yaml`（开发）+ `compose.prod.yaml`（生产），4 个容器：PC 前端 + 移动端 + 后端 + MySQL。`deploy.sh` 通过 `-f` 参数区分环境。

- 单机部署场景，4 容器无需 Pod/Service/Ingress 抽象层
- `deploy.resources` 直接限制 CPU/内存（`128M/0.5CPU` ~ `512M/1CPU`）
- K8s 对这一规模是严重过度设计

##### 决策 9：`bus_bill_batch` — 孤立的 Entity/Mapper, 表从未被创建

**可核实的事实**：
- `BillBatch.java`（Entity，`@TableName("bus_bill_batch")`）存在于 `wisdom-pojo/entity/`
- `BillBatchMapper.java`（接口，继承 `BaseMapper<BillBatch>`）存在于 `wisdom-server/mapper/`
- `FinanceServiceImpl:43` 声明 `@Autowired private BillBatchMapper billBatchMapper;`
- 全文搜索 `billBatch` 在 FinanceServiceImpl 中无任何方法调用——该字段仅声明从未使用
- 测试类 `FinanceServiceImplTest:20` import 了 `BillBatchMapper`，测试中 `@MockBean` 声明但无测试用例覆盖
- **`01-init.sql` 中 12 个 CREATE TABLE 不含 `bus_bill_batch`**
- **唯一引用在 `02-demo-data.sql:63`**：`INSERT INTO bus_bill_batch (...) VALUES (...)` ——但该表从未被 CREATE，Docker 启动执行 SQL 脚本时此 INSERT 会失败
- MyBatis-Plus 不会自动建表（`application.yml` 中无 `ddl-auto` 配置）

**结论**：`bus_bill_batch` 是孤立的 Java 代码——Entity 和 Mapper 存在，但 SQL DDL 缺失，数据库中没有这张表。Demo data 的 INSERT 语句是无效的。

##### 决策 10：AI 模块为什么是 Stub 占位？

`AIController.java`（40 行，3 个接口）+ `AIServiceImpl.java`（Stub 实现，每个方法返回固定字符串）。接口骨架完整（Swagger 注解 + DTO 参数），待接入真实 AI SDK。

```
任何 HTTP 请求
    │
    ├─ URL startsWith 白名单路径?
    │   (AuthenticationFilter.isWhiteList() 使用 uri.startsWith(path))
    │   ✓ /api/v1/auth/login        → 跳过鉴权 (匹配该前缀)
    │   ✓ /api/v1/auth/register     → 跳过鉴权 (匹配该前缀)

    │
    └─ ✗ 不在白名单中:
        │
        ├─ JwtAuthenticationFilter (@Order=1)  ← 先执行, 不阻断
        │   └─ Authorization 头有 Bearer token → 验签 → 写入 BaseContext
        │   └─ 无论 token 是否有效 → 都放行
        │
        └─ AuthenticationFilter (@Order=2)     ← 后执行, 可阻断
            └─ handlerMapping.getHandler(request) → 反射查 @LoginRequired
                ├─ 方法无 @LoginRequired → 放行（无需认证的接口）
                └─ 方法有 @LoginRequired:
                    ├─ Bearer token 有效 → 写入 BaseContext → 放行
                    └─ Bearer token 无效/缺失 → 返回 {"code":401,"msg":"UNAUTHORIZED"}
```

用 `RequestMappingHandlerMapping.getHandler()` 而不是路径前缀匹配的原因：路径前缀匹配需要维护一个"需要认证的路径列表"，与 Controller 上的注解脱节。动态反射方式让鉴权规则与接口定义同在一处——在方法上加 `@LoginRequired` 即表示需要认证，不加即公开。

#### 代码解析：HTTP 封装 + Pinia Store + 路由守卫

**http.ts — 原生 fetch 封装**（`services/http.ts`，114 行）

项目未使用 Axios（虽然 `package.json` 声明了依赖），而是基于原生 `fetch` 实现 HTTP 客户端：

```typescript
const TOKEN_KEY = 'wisdompm-token'

export function getToken() {
  if (typeof window === 'undefined') {
    return ''
  }
  return window.sessionStorage.getItem(TOKEN_KEY) || ''
}

export function setToken(token: string) {
  if (typeof window !== 'undefined') {
    window.localStorage.removeItem(TOKEN_KEY)
    window.sessionStorage.setItem(TOKEN_KEY, token)
  }
}

export function clearToken() {
  if (typeof window !== 'undefined') {
    window.sessionStorage.removeItem(TOKEN_KEY)
    window.localStorage.removeItem(TOKEN_KEY)
  }
}
```

核心 `request<T>()` 函数：

```typescript
export async function request<T>(path: string, options: RequestOptions = {}) {
  const { body, headers, query, skipJson, ...rest } = options
  const token = getToken()
  const isFormData = typeof FormData !== 'undefined' && body instanceof FormData
  const nextHeaders = new Headers(headers)

  if (token) {
    nextHeaders.set('Authorization', `Bearer ${token}`)
  }
  if (!skipJson && body !== undefined && !isFormData) {
    nextHeaders.set('Content-Type', 'application/json')
  }

  const started = performance.now()
  let success = false

  try {
    const response = await fetch(buildUrl(path, query), {
      ...rest,
      headers: nextHeaders,
      body:
        body === undefined
          ? undefined
          : skipJson || isFormData
            ? (body as BodyInit)
            : JSON.stringify(body),
    })

    if (AUTH_EXPIRED_CODES.has(response.status)) {
      handleAuthExpired()
    }

    const result = (await response.json()) as ApiResponse<T>

    if (!response.ok) {
      throw new Error(result.msg || `请求失败: ${response.status}`)
    }

    if (result.code !== 200) {
      if (AUTH_EXPIRED_CODES.has(result.code)) {
        handleAuthExpired()
      }
      throw new Error(result.msg || '接口返回异常')
    }

    success = true
    return result.data
  } finally {
    const duration = performance.now() - started
    monitor.recordRequest(duration, success)
  }
}
```

`handleAuthExpired()` 通过 `clearToken()` + `window.dispatchEvent(new CustomEvent('wisdompm:auth-expired'))` 解耦——http.ts 不需要知道 Pinia Store 的存在。`buildUrl()` 自动过滤 `null/undefined/''` 参数。`finally` 块调 `monitor.recordRequest()` 记录每次请求的成功率和耗时。返回值 `result.data` 而非完整 `result`，调用方直接用业务数据。所有 token 读写方法均有 `typeof window === 'undefined'` 守卫，兼容 SSR 场景。

**路由守卫**（`router/index.ts`）——`beforeEach` 中判断是否已登录：

```typescript
router.beforeEach((to, _from) => {
  const appStore = useAppStore()
  const isAuthPage = to.path === '/login' || to.path === '/register'

  if (!appStore.isAuthenticated && !isAuthPage) {
    return { path: '/login', query: { redirect: to.fullPath } }
  }

  if (appStore.isAuthenticated && isAuthPage) {
    return { path: '/dashboard' }
  }

  document.title = `${String(to.meta.title ?? '智慧物业管理系统')} | WisdomPM`
})
```

Pinia Store 的 `isAuthenticated` 基于 `sessionStorage` 持久化。关闭标签页自动清除——这正是选择 `sessionStorage` 而非 `localStorage` 的原因：标签页关闭 = 登出，无需手动调用 logout。

**Pinia Store**（`stores/app.ts`）——单一 Store 架构，169 行覆盖认证 + 侧边栏 + 用户信息：

```typescript
export const useAppStore = defineStore('app', () => {
  const savedState = loadState()
  const savedProfile = loadProfile()

  const sidebarCollapsed = ref(savedState?.compactMode ?? false)
  const isAuthenticated = ref(savedState?.isAuthenticated ?? false)

  async function loginByCredential(payload: LoginPayload) {
    const token = await authApi.login(payload)
    setToken(token)
    const profile = await fetchCurrentUser()
    return profile
  }

  function enablePreviewSession() {
    if (!import.meta.env.DEV) {
      return
    }
    isAuthenticated.value = true
    currentUserProfile.value = {
      id: 1,
      username: 'preview',
      realName: '预览管理员',
      phone: '13800000000',
      email: 'preview@wisdompm.local',
      status: 1,
      roleId: 1,
      roleName: '超级管理员',
    }
    persistProfile()
    persistState()
  }

  function persistState() {
    if (typeof window === 'undefined') return
    window.localStorage.removeItem(STORAGE_KEY)
    window.sessionStorage.setItem(STORAGE_KEY, JSON.stringify({
```

`sessionStorage` 存序列化的 `{isAuthenticated, compactMode}` + `UserInfo`。每次写前先 `localStorage.removeItem()` 清掉可能存在的旧值，确保数据只存 `sessionStorage`。

#### 代码解析：移动端 Axios 封装

移动端使用 Axios（与 PC 端的原生 `fetch` 不同），核心差异在于 Axios 的拦截器机制：

**http.ts — Axios 实例 + 双拦截器**（`services/http.ts`，51 行）

```typescript
const http = axios.create({
    baseURL: '/api/v1',
    timeout: 15000,
})

http.interceptors.request.use((config) => {
    const token = sessionStorage.getItem(TOKEN_KEY)
    if (token) {
        config.headers.Authorization = `Bearer ${token}`
    }
    return config
})

http.interceptors.response.use(
    (response) => {
        const result = response.data as ApiResponse<unknown>
        if (result.code !== 200) {
            if (result.code === 401 || result.code === 403) {
                sessionStorage.removeItem(TOKEN_KEY)
                window.location.hash = '#/login'
            }
            showToast(result.msg || '请求失败')
            return Promise.reject(new Error(result.msg))
        }
        return result.data as any
    },
    (error) => {
        showToast('网络异常，请稍后重试')
        return Promise.reject(error)
    },
)
```

**PC vs 移动端 HTTP 层对比**：

|         | PC 端                                        | 移动端                                   |
|:------- |:------------------------------------------- |:------------------------------------- |
| 库       | 原生 `fetch`                                  | Axios                                 |
| baseURL | 相对路径（经 Nginx 代理）                            | `/api/v1`（Axios 预设）                   |
| 超时      | 无默认（浏览器默认 ~90s）                             | 15000ms                               |
| 401 处理  | `CustomEvent('wisdompm:auth-expired')` 事件解耦 | 直接 `window.location.hash = '#/login'` |
| 错误提示    | 仅 throw Error                               | Vant `showToast()` + throw            |
| 参数清洗    | `buildUrl()` 自动过滤 undefined/null            | 无（Axios 默认不做过滤）                       |
| 请求监控    | `monitor.recordRequest()`                   | 无                                     |

### 6.2 规范

| 规范项 | 约定 | 说明 |
|:------|:-----|:----|
| 表名前缀 | `sys_`（系统表）/ `bus_`（业务表） | 快速区分表类型 |
| 字段命名 | `snake_case`（小写+下划线） | 如 `create_time`, `bill_month` |
| 主键 | `BIGINT AUTO_INCREMENT` | 统一自增 ID |

| 字符集 | `utf8mb4` | `my.cnf` 统一配置 |

### 6.3 迁移管理

`sql/` 6 个脚本按文件名序执行，Docker 首次启动自动运行（通过 `/docker-entrypoint-initdb.d` 挂载）。DDL 必须幂等，禁止修改旧脚本。目录内含 `my.cnf` 配置 MySQL 字符集为 `utf8mb4`。

已知 SQL 兼容性问题：
- `02-demo-data.sql` 列名不兼容：`bus_bill` 的 INSERT 使用旧版列名 (`type`, `pay_status`, `deadline`)，而 `01-init.sql` 创建的表使用当前 schema (`fee_type`, `status`, `due_date`)。同样 `bus_payment_record` 使用旧版列名 (`trx_no`, `pay_type`, `operator`)，当前 schema 为 (`payment_no`, `pay_method`)。Docker 首次启动时 demo data 的 INSERT 会全部失败。
- 迁移脚本依赖失效：`04-migrate-house-names.sql` 引用 `02-demo-data.sql` 的数据（其 INSERT 已失效）；`05-migrate-extra-house-names.sql` 引用的房屋 ID (24-40) 在任何脚本中都未播种。
- 以上问题不影响 DDL（表结构创建正常），仅影响 demo 测试数据。
