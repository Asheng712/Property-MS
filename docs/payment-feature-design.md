# 缴费功能修改设计方案

## 1. 设计目标

缴费功能按照以下业务链路设计：

```text
合同 -> 生成账单 -> 用户提交支付记录 -> 管理员核销 -> 更新账单状态
```

核心规则：

- 费用类型固定为租金、买房金额、押金、物业费。
- 不允许部分缴费。
- 一个支付记录允许对应多个账单。
- 管理员核销后才更新账单为已缴费。
- 核销后允许撤销。
- 物业费调价从设置时的下一个月开始生效。
- 历史账单金额不因物业费调价而重新计算。

## 2. 费用类型

费用类型建议在 `wisdom-common.enumeration` 中定义枚举，数据库使用 `TINYINT` 存储。

| 类型值 | 费用类型 |
| :--- | :--- |
| 1 | 租金 |
| 2 | 买房金额 |
| 3 | 押金 |
| 4 | 物业费 |

后端业务逻辑应基于枚举值判断费用类型，前端只负责展示对应中文名称。

## 3. 账单设计

账单表示系统对用户的应收费用。

### 3.1 账单表建议字段

表名建议：`bill`

| 字段名 | 类型 | 说明 |
| :--- | :--- | :--- |
| id | BIGINT | 主键 |
| user_id | BIGINT | 用户 ID |
| house_id | BIGINT | 房屋 ID |
| contract_id | BIGINT | 合同 ID |
| fee_type | TINYINT | 费用类型 |
| bill_no | VARCHAR | 账单编号 |
| bill_month | VARCHAR | 账单月份，如 `2026-07` |
| amount | DECIMAL(12,2) | 应缴金额 |
| status | TINYINT | 账单状态 |
| due_date | DATE | 截止日期 |
| remark | VARCHAR | 备注 |
| create_time | DATETIME | 创建时间 |
| update_time | DATETIME | 更新时间 |

### 3.2 账单状态

| 状态值 | 状态 | 说明 |
| :--- | :--- | :--- |
| 0 | 待缴费 | 用户尚未提交缴费 |
| 1 | 待核销 | 用户已提交支付记录，等待管理员核销 |
| 2 | 已缴费 | 管理员已核销 |
| 3 | 已撤销 | 账单被撤销 |
| 4 | 已作废 | 账单不可继续缴纳 |

由于不允许部分缴费，账单表不需要维护复杂的 `paid_amount`、`unpaid_amount` 字段。用户提交的支付金额必须等于所选账单总金额。

## 4. 支付记录设计

支付记录表示用户的一次缴费行为。由于一个支付记录可以对应多个账单，建议拆分为支付主表和支付明细表。

### 4.1 支付主表

表名建议：`payment_record`

| 字段名 | 类型 | 说明 |
| :--- | :--- | :--- |
| id | BIGINT | 主键 |
| payment_no | VARCHAR | 支付记录编号 |
| user_id | BIGINT | 缴费用户 ID |
| amount | DECIMAL(12,2) | 本次支付总金额 |
| pay_method | TINYINT | 支付方式 |
| pay_time | DATETIME | 用户支付时间 |
| status | TINYINT | 支付状态 |
| voucher_url | VARCHAR | 支付凭证地址 |
| remark | VARCHAR | 用户备注 |
| verify_user | BIGINT | 核销管理员 ID |
| verify_time | DATETIME | 核销时间 |
| cancel_user | BIGINT | 撤销管理员 ID |
| cancel_time | DATETIME | 撤销时间 |
| cancel_reason | VARCHAR | 撤销原因 |
| create_time | DATETIME | 创建时间 |
| update_time | DATETIME | 更新时间 |

### 4.2 支付状态

| 状态值 | 状态 | 说明 |
| :--- | :--- | :--- |
| 0 | 待核销 | 用户已提交，管理员未确认 |
| 1 | 已核销 | 管理员确认到账 |
| 2 | 已驳回 | 管理员驳回支付记录 |
| 3 | 已撤销 | 已核销记录被撤销 |

### 4.3 支付明细表

表名建议：`payment_record_bill`

| 字段名 | 类型 | 说明 |
| :--- | :--- | :--- |
| id | BIGINT | 主键 |
| payment_record_id | BIGINT | 支付记录 ID |
| bill_id | BIGINT | 账单 ID |
| amount | DECIMAL(12,2) | 对应账单金额 |
| create_time | DATETIME | 创建时间 |
| update_time | DATETIME | 更新时间 |

## 5. 物业费配置设计

物业费单价不应写死在代码中。建议新增物业费配置表。

表名建议：`property_fee_config`

| 字段名 | 类型 | 说明 |
| :--- | :--- | :--- |
| id | BIGINT | 主键 |
| unit_price | DECIMAL(12,2) | 每平方米每月单价 |
| effective_month | VARCHAR | 生效月份，如 `2026-07` |
| status | TINYINT | 状态，0 禁用，1 启用 |
| create_time | DATETIME | 创建时间 |
| update_time | DATETIME | 更新时间 |

默认物业费单价为：

```text
2.00 元 / 平方米 / 月
```

调价规则：

- 管理员设置新价格后，从下一个月开始生效。
- 例如 2026-06-08 设置为 3.00 元，则 `effective_month = 2026-07`。
- 生成物业费账单时，查询账单月份对应的生效单价。
- 账单生成后，金额固化到账单表中。
- 历史账单不重新计算。

物业费金额计算：

```text
物业费金额 = 房屋面积 * 当月生效单价
```

金额计算必须使用 `BigDecimal`，数据库金额字段必须使用 `DECIMAL(12,2)`。

## 6. 合同生成账单规则

账单应根据已有合同生成。

### 6.1 租赁合同

租赁合同可生成以下账单：

- 押金账单。
- 租金账单。
- 物业费账单。

### 6.2 购房合同

购房合同可生成以下账单：

- 买房金额账单。
- 物业费账单。

### 6.3 生成时机

建议先实现手动生成账单能力，后续再扩展定时任务。

- 合同生效时生成一次性账单，如押金、买房金额。
- 租金按合同周期生成，优先按月生成。
- 物业费按月生成。
- 每张账单生成时固化当前应收金额。

## 7. 移动端用户缴费流程

移动端用户流程：

1. 用户查看自己的待缴账单。
2. 用户选择一张或多张账单。
3. 系统计算账单合计金额。
4. 用户提交支付信息或支付凭证。
5. 后端创建支付记录和支付明细。
6. 后端将关联账单状态更新为待核销。

后端提交缴费时必须校验：

- 所选账单必须属于当前登录用户。
- 所选账单状态必须是待缴费。
- 支付金额必须等于所选账单金额合计。
- 不允许部分缴费。
- 同一账单不能被重复提交支付记录。

## 8. Web 管理端核销流程

管理员核销流程：

1. 管理员查看待核销支付记录列表。
2. 管理员查看支付记录对应的账单明细。
3. 管理员确认实际到账后执行核销。
4. 后端更新支付记录为已核销。
5. 后端同步更新关联账单为已缴费。

核销操作必须使用事务，支付记录和账单状态要同时成功或同时失败。

## 9. 核销撤销流程

核销后允许撤销，但不应删除支付记录和账单数据。

撤销已核销支付记录时：

- 支付记录状态更新为已撤销。
- 记录撤销管理员、撤销时间、撤销原因。
- 关联账单状态从已缴费回退为待缴费。

推荐回退为待缴费，而不是待核销。原因是撤销核销通常表示该笔到账不能继续作为有效缴费，需要用户重新提交或管理员重新处理。

撤销操作同样必须使用事务。

## 12. 事务与数据一致性

以下操作必须使用事务：

- 用户提交支付记录，并将账单状态更新为待核销。
- 管理员核销支付记录，并将账单状态更新为已缴费。
- 管理员撤销核销，并将账单状态回退为待缴费。

关键一致性规则：

- 支付记录金额必须等于关联账单金额合计。
- 支付记录明细中的账单金额应与账单表金额一致。
- 待核销或已缴费账单不能再次发起缴费。
- 已撤销支付记录不能再次核销。
- 已核销支付记录不能重复核销。

## 13. 推荐实现顺序

1. 新增枚举、Entity、DTO、VO。
2. 新增数据库表结构和初始化默认物业费配置。
3. 实现账单生成接口。
4. 实现移动端查询账单和提交缴费接口。
5. 实现 Web 端待核销列表和核销接口。
6. 实现核销撤销接口。
7. 实现物业费调价接口。
8. 同步前端 `types` 和 `services`。
9. 完成 Web 管理端页面。
10. 完成移动端用户页面。

## 14. 总结

本方案将缴费功能拆分为账单、支付记录、支付明细、物业费配置四个核心对象。

系统根据合同生成应收账单，用户可一次性选择多张账单全额缴费。缴费提交后账单进入待核销状态，管理员核销后账单才变为已缴费。核销支持撤销，撤销后账单回退为待缴费。物业费调价从下月账单开始生效，历史账单金额保持不变。
