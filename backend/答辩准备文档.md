### 2.3 认证鉴权体系 (双Filter架构) ★核心亮点★

```
请求 → [JwtAuthenticationFilter @Order=1] → [AuthenticationFilter @Order=2] → Controller

JwtAuthenticationFilter — "被动提取者" (JwtAuthenticationFilter.java:22-43):
  ├─ 从 Authorization Header 提取 "Bearer xxx"
  ├─ 验证Token成功 → 解析userId → BaseContext.setCurrentId(userId)
  ├─ 验证Token失败 → 静默跳过（不拦截请求!）
  └─ finally: BaseContext.removeCurrentId() ← 防止内存泄漏

AuthenticationFilter — "主动守卫者" (AuthenticationFilter.java:47-93):
  ├─ 检查白名单: /api/v1/auth/login, /register, /swagger-ui/** → 直接放行
  ├─ 反射获取 HandlerMethod → 检查 @LoginRequired 注解
  │    ├─ 无 @LoginRequired → 公开接口，放行
  │    └─ 有 @LoginRequired → 校验Token有效性
  │         ├─ 有效 → 放行
  │         └─ 无效/缺失 → 返回401 JSON (code=401, msg="UNAUTHORIZED"/"INVALID_TOKEN")
  └─ finally: BaseContext.removeCurrentId()
```

**核心源码** — 注解驱动的鉴权判断 (`AuthenticationFilter.java:109-124`):
```java
private boolean requiresLogin(HttpServletRequest request) {
    try {
        HandlerExecutionChain handlerChain = handlerMapping.getHandler(request);
        if (handlerChain == null) return false;  // 静态资源

        if (handlerChain.getHandler() instanceof HandlerMethod handlerMethod) {
            Method method = handlerMethod.getMethod();
            return method.isAnnotationPresent(LoginRequired.class);  // ← 注解驱动!
        }
        return false;
    } catch (Exception e) {
        log.debug("Handler mapping failed for: {}", request.getRequestURI(), e);
        return false;  // fail-open: 异常时放行（安全性偏宽松）
    }
}
```

**设计评价**:
- ✅ 职责分离: Token解析 vs 鉴权决策，各自独立
- ✅ 注解驱动: `@LoginRequired` 声明式鉴权，无需维护URL配置表
- ✅ 白名单模式: 默认公开，按需保护（符合最小惊讶原则）
- ⚠️ Token被两个Filter各解析一次，存在微小性能开销
- ⚠️ `BaseContext.threadLocal` 为 `public` 字段，封装性可改进
- ⚠️ HandlerMapping查找失败时返回false（fail-open），安全性偏宽松

### 2.4 ThreadLocal 用户上下文传递

```java
// BaseContext.java — 线程级用户ID存储
public class BaseContext {
    public static ThreadLocal<Long> threadLocal = new ThreadLocal<>();
    public static void setCurrentId(Long id) { threadLocal.set(id); }
    public static Long getCurrentId() { return threadLocal.get(); }
    public static void removeCurrentId() { threadLocal.remove(); }
}
```

**ThreadLocal 的必要性**:
1. Tomcat为每个请求分配一个线程，ThreadLocal天然做到请求隔离
2. 避免在 Controller → Service → Mapper 每一层传递 `userId` 参数
3. Filter的finally块保证线程归还线程池前清理，防止内存泄漏和数据串扰
4. 若使用 `@Async` 异步方法，需显式传递ThreadLocal值（InheritableThreadLocal或手动传参）

public ResponseEntity<Result<Void>> businessExceptionHandler(BusinessException ex) {
    HttpStatus status = switch (ex.getCode()) {
        case 401 -> HttpStatus.UNAUTHORIZED;
        case 403 -> HttpStatus.FORBIDDEN;
        case 404 -> HttpStatus.NOT_FOUND;
        case 400 -> HttpStatus.BAD_REQUEST;

### 2.8 JWT 实现详解

```java
// JwtTokenUtil.java — 构造器注入密钥和过期时间
@Component
public class JwtTokenUtil {
    private final SecretKey secretKey;
    private final long expiration;

    public JwtTokenUtil(
            @Value("${jwt.secret}") String secret,
            @Value("${jwt.expiration:86400000}") long expiration  // 默认24小时
    ) {
        this.secretKey = Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
        this.expiration = expiration;
    }

    public String generateToken(Long userId, String username) {
        Map<String, Object> claims = new HashMap<>();
        claims.put("userId", userId);
        claims.put("username", username);
        return Jwts.builder()
                .claims(claims)
                .subject(username)
                .expiration(new Date(System.currentTimeMillis() + expiration))
                .signWith(secretKey)        // HMAC-SHA 签名
                .compact();
    }

    public Long getUserIdFromToken(String token) { ... }
    public boolean validateToken(String token) { ... }
}
```

**Token 存储的信息**: userId + username + 过期时间
**不存储**: 密码、角色权限 — 敏感信息在服务端按需查询

### 3.4 HTTP 请求层 (核心实现)

```typescript
// services/http.ts — 基于 Fetch API 的封装
export async function request<T>(path: string, options: RequestOptions = {}) {
  const { body, headers, query, skipJson, ...rest } = options
  const token = getToken()
  const nextHeaders = new Headers(headers)

  // 自动携带 Token
  if (token) nextHeaders.set('Authorization', `Bearer ${token}`)

  // 自动设 JSON Content-Type
  if (!skipJson && body !== undefined && !isFormData) {
    nextHeaders.set('Content-Type', 'application/json')
  }

  const response = await fetch(buildUrl(path, query), {
    ...rest, headers: nextHeaders,
    body: body === undefined ? undefined
      : skipJson || isFormData ? (body as BodyInit) : JSON.stringify(body),
  })

  // 401/403 → 自动清理Token + 派发事件
  if (AUTH_EXPIRED_CODES.has(response.status)) handleAuthExpired()

  const result = await response.json()
  if (result.code !== 200) throw new Error(result.msg || '接口返回异常')

  return result.data  // 解包：直接返回 data 字段
}
```

**query参数过滤逻辑** (`http.ts:48-58`):
```typescript
// null/undefined/'' 自动从URL中排除
Object.entries(query).forEach(([key, value]) => {
  if (value === undefined || value === null || value === '') return
  url.searchParams.set(key, String(value))
})
```

**Token存储策略**:
- 使用 `sessionStorage`（关闭标签页即清除），非 `localStorage`

  async function loginByCredential(payload) {

### 5.1 设计模式应用全景

| 模式 | 应用位置 | 具体代码 |
|------|---------|---------|
| **单例 (Singleton)** | Spring Bean | 所有 @Service, @Component, @Configuration |
| **策略 (Strategy)** | 账单金额计算 | `switch(feeType)`: RENT→合同租金, PURCHASE→购房款, DEPOSIT→押金, PROPERTY→面积×单价 |
| **模板方法 (Template)** | Service 接口+实现 | `AssetService` 接口 → `AssetServiceImpl` |
| **责任链 (Chain of Resp.)** | Filter 链 | JwtAuthenticationFilter(解析) → AuthenticationFilter(鉴权) |
| **观察者 (Observer)** | Token过期通知 | `window.dispatchEvent(new CustomEvent('wisdompm:auth-expired'))` |
| **外观 (Facade)** | Controller 聚合Service | DashboardController 聚合 SystemService |
| **工厂方法 (Factory)** | 枚举 fromCode | `FeeTypeEnum.fromCode(code)` → 枚举实例 |
| **静态工厂方法** | BusinessException | `BusinessException.unauthorized()` vs `new BusinessException(401, ...)` |

### 5.2 为什么用 BCrypt 而不是 MD5？

- MD5 已被证明不安全: 彩虹表攻击、碰撞攻击（两个不同输入产生相同哈希）
- BCrypt 自带随机 salt（每个密码自动生成不同的盐值）+ 可调节计算成本(work factor)
- Spring Security 官方推荐的密码编码器
- 即使数据库泄露，攻击者也无法批量破解密码

### 5.3 为什么用 MyBatis-Plus 而不是 JPA/Hibernate？

- **复杂查询**: LambdaQueryWrapper 提供了类型安全的动态SQL构建
- **分页**: `PaginationInnerInterceptor` 一行配置搞定物理分页
- **国内生态**: 中文文档完善，社区活跃
- **灵活性**: 本项目需要账单聚合、看板统计等灵活查询，MyBatis-Plus比Hibernate的HQL更直观

### 5.4 为什么用 sessionStorage 而不是 localStorage 存 Token？

- **sessionStorage**: 关闭浏览器标签页即清除，生命周期短，安全性高
- **localStorage**: 持久化存储，XSS 攻击后Token长期暴露
- 对于物业管理系统这种后台应用，用户每次打开都会重新登录或通过 sessionStorage 恢复，体验影响小

### 5.5 为什么当前没有用 Redis？

作为演示/教学项目，当前规模不需要:
- 用户量和并发低，直连MySQL足够
- Token存在sessionStorage，不需要服务端状态
- 数据量小

未来引入Redis的场景: Token黑名单(主动登出)、看板缓存、限流计数器、分布式Session。

### 5.6 为什么 Spring Boot 3.2 + Java 17？

- Spring Boot 3.x 强制要求 Java 17，带来虚拟线程、GraalVM原生编译、Jakarta EE 9+标准化
- Java 17 是 LTS 版本，长期安全支持
- Records、Pattern Matching、switch表达式等新语法特性让代码更简洁

## 七、代码亮点与不足

### 7.1 代码亮点 ⭐

**1. 双 Filter 分离关注点**
JwtAuthenticationFilter 只解析Token，AuthenticationFilter 只做鉴权决策——单一职责原则的完美体现。

**2. 注解驱动的权限控制**
`@LoginRequired` + Spring HandlerMapping 反射 → 声明式鉴权，无需维护URL配置表。

**3. 统一异常体系**
`BusinessException` 提供静态工厂方法 (`unauthorized()`, `forbidden()`, `notFound()`, `badRequest()`) → 调用方代码语义清晰，错误码统一管理。

**4. 账单防重复生成**
同房屋+同费类型+同月份+非作废 → 联合唯一键语义，防止重复计费。

**5. ThreadLocal 清理保障**
两个 Filter 都在 finally 块调用 `removeCurrentId()` → 线程归还线程池前清理，防止内存泄漏和数据串扰。

**6. 接口与实现分离 (AI模块)**
`AIService` 接口在 wisdom-common → 模拟实现 AIServiceImpl 在 wisdom-server → 未来替换AI引擎零侵入。

**7. 前端监控体系**
自建 monitor (请求指标) + logger (结构化日志) → 为生产环境可观测性打基础。

**8. 枚举统一管理状态值**
所有状态码都有对应的 Enum (BillStatusEnum, PaymentStatusEnum, FeeTypeEnum, PayMethodEnum) → 消除魔法数字。

**9. 事务管理**
关键操作 (@Transactional(rollbackFor = Exception.class)): 缴费提交、核销、审批 → 保证ACID。

**10. 用户资料级联同步**
updateProfile 变更姓名/手机时 → 自动同步房产ownerName、合同tenantName、投诉source → 数据一致性。

**11. 审批+自动生成账单**
购买/租赁审批通过 → 自动创建合同 + 生成账单 → 业务流程自动化，减少人工遗漏。

**12. 响应式布局**
AdminLayout 支持侧边栏折叠(紧凑模式) + 移动端 Drawer 抽屉 → 多屏幕适配。

### 7.2 可改进之处 ⚠️

**1. Token 重复解析**
两个Filter各解析一次JWT，可通过 `HttpServletRequest.setAttribute` 传递解析结果避免重复。

**2. BaseContext 封装性**
`threadLocal` 字段为 `public` → 应改为 `private`，强制通过 getter/setter 访问。

**3. 缺少输入校验**
部分Service方法未做充分的参数校验，依赖数据库约束兜底 → Controller层应统一使用 `@Valid` + Jakarta Validation。

**4. 缺少接口限流**
无 Rate Limiting → 生产环境易被高频请求攻击。可引入 Guava RateLimiter 或 Redis 滑动窗口。

**5. 密码强度校验缺失**
注册时未校验密码复杂度（长度/大小写/特殊字符）→ 弱密码风险。

**6. 前端类型历史遗留**
`types/index.ts` 含大量 `Legacy*` 类型定义（LegacyAssetTreeNode 等）→ 代码已不使用但未清理。

**7. 导入导出为模拟实现**
`SystemServiceImpl.importAssets()` 和 `exportFinanceReport()` 仅创建 FileTask 记录并标记SUCCESS → 未实现真实文件解析/生成。

**8. 白名单硬编码**
`AuthenticationFilter.WHITE_LIST` 字符串数组 → 新增公开接口需改代码。可改为读取注解或配置文件。

**9. BeanUtils.copyProperties 性能**
基于反射 → 大量数据转换时性能差。可引入 MapStruct（编译期代码生成，零反射）。

**10. 看板全表查询**
`repairMapper.selectList(null)` → 数据量大时性能差。应加LIMIT或分页+缓存。

**11. 费用计算 switch 可维护性**
`calculateBillAmount()` 使用 switch → 新增费用类型需改代码。可改为策略模式。

**12. 合同模板硬编码**
`ContractServiceImpl.saveOrUpdateContract()` 中 deposit>0 判定租房 vs 买房 → 逻辑耦合在代码中，不够灵活。

### Q2: 双 Filter 认证架构的设计思路？为什么不用 Spring Security？

**答**:

**设计思路**:
- `JwtAuthenticationFilter` (@Order=1): "被动提取者"——只解析Token，不拦截请求。即使Token无效也静默跳过
- `AuthenticationFilter` (@Order=2): "主动守卫者"——检查白名单 + @LoginRequired注解 + Token有效性，三者联合决策

这种**职责分离**的设计使每个Filter只做一件事，代码清晰易维护。

**为什么不用 Spring Security**:
1. **复杂度**: Spring Security 的 FilterSecurityInterceptor + AuthenticationManager + SecurityContextHolder 体系对于本项目权限模型（两级：管理员/普通用户）来说过于重量级
2. **可控性**: 自研Filter约200行代码，完全可控，调试方便。Spring Security的"魔法行为"（自动生成登录页、CSRF保护等）需要额外配置关闭
3. **学习成本**: 团队对Spring Security不熟悉的情况下，自研方案更容易理解和维护
4. **声明式鉴权**: `@LoginRequired` 注解提供了与 `@PreAuthorize` 类似的声明式体验，但更轻量

**如果角色增多到5+种**，建议迁移到 Spring Security + RBAC。

### Q3: ThreadLocal 在 Web 应用中的使用方式和注意事项？

**答**:

**使用方式** (本项目):
```java
// Filter中: 解析Token → 存入ThreadLocal
BaseContext.setCurrentId(userId);
try { chain.doFilter(request, response); }
finally { BaseContext.removeCurrentId(); }  // ← 关键!
```

**Service层使用**:
```java
Long currentUserId = BaseContext.getCurrentId();  // 无需传参
```

**注意事项**:
1. **必须清理**: Tomcat使用线程池，线程会复用。不清理会导致：
   - 内存泄漏: ThreadLocal的Entry强引用无法被GC
   - 数据串扰: 下一个请求拿到上一个请求的用户ID（严重安全漏洞）
2. **子线程传递**: `@Async` 或自定义线程池的场景，ThreadLocal不会自动传递。解决方案：InheritableThreadLocal 或手动将userId传递给异步方法
3. **不用存大对象**: ThreadLocal生命周期与线程绑定，大量数据可能导致OOM

### Q4: JWT Token 的安全性如何保证？有什么局限？

**答**:

**安全措施**:
1. **签名算法**: HMAC-SHA (jjwt 0.12.5)，密钥通过 `${jwt.secret}` 环境变量注入，不硬编码
2. **过期时间**: 默认24小时，通过 `jwt.expiration` 可配置
3. **不存敏感信息**: Token中只存 userId + username + exp，不存密码/角色/权限
4. **每次验证**: 每次请求都验签+验过期，防止篡改和重放
5. **前端存储**: sessionStorage（非localStorage），关闭标签页即失效
6. **HTTPS传输**: 生产环境强制HTTPS，防止Token被网络嗅探

**局限性**:
1. 无Refresh Token → Token过期需重新登录（单体应用的常见简化）
2. 无Token黑名单 → 无法主动失效某个Token（如管理员踢出用户、密码重置后旧Token仍有效）
3. Token一旦签发就独立有效 → 无法实时感知用户状态变更（如被封禁）

**改进方向**: 引入Redis维护Token黑名单，或使用OAuth2.0授权码模式 + Refresh Token。

### Q5: 为什么要分 DTO、VO、Entity 三层对象？

**答**:

| 对象 | 用途 | 约束 |
|------|------|------|
| **Entity** | 与数据库表一一映射 | 仅限Mapper层使用，禁止返回到Controller层 |
| **DTO** | 接收前端请求参数 | 可含 @Valid 校验注解，可组合多个Entity字段 |
| **VO** | 返回给前端的视图对象 | 可脱敏(不返password)、可组合(含houseName)、可翻译(status→中文) |

**实际例子 — RepairVO vs Repair Entity**:
```java
// Repair Entity: 只有基本字段 (repairNo, houseId, status=0/1/2, priority=1/2/3)
// RepairVO: 额外组合了 houseName, workerName(真实姓名),
//            statusText("待处理"/"处理中"), priorityText("普通"/"紧急")
```

**核心价值**:
- 解耦数据库结构与API契约
- 数据库字段变更（如重命名列）不影响前端
- 前端展示需求变化（如增加计算字段）不波及数据库
- 安全: VO可以过滤敏感字段（如User.password永远不会出现在UserVO中）

### Q6: 账单防重复生成是怎么做的？能防御并发吗？

**答**:

**防重复逻辑** (`FinanceServiceImpl.java:98-103`):
```java
LambdaQueryWrapper<Bill> dupQuery = new LambdaQueryWrapper<>();
dupQuery.eq(Bill::getHouseId, house.getId())
        .eq(Bill::getFeeType, feeType)
        .eq(Bill::getBillMonth, billMonth)
        .ne(Bill::getStatus, BillStatusEnum.VOIDED.getCode());
if (billMapper.selectCount(dupQuery) > 0) continue;
```

通过 **(houseId, feeType, billMonth, status≠4)** 组合去重。

**并发问题分析**:
当前 `generateBills()` 有 `@Transactional`，但在默认隔离级别(READ_COMMITTED)下，两个并发请求的 SELECT 都查不到记录，然后都执行 INSERT → 产生重复账单。

**解决方案**:
1. **数据库唯一索引** (推荐): 在 (house_id, fee_type, bill_month, status) 加联合唯一索引，数据库层面兜底
2. **分布式锁**: Redis SETNX 或 `SELECT ... FOR UPDATE` 悲观锁
3. **幂等键**: 前端生成 requestId，后端用唯一索引防重

### Q10: 如何保证支付核销的数据一致性？

**答**: 使用 Spring `@Transactional` 保证ACID：

```java
@Transactional(rollbackFor = Exception.class)
public void verifyPayment(Long id) {
    // 1. 更新支付记录: status→已核销, verifyUser, verifyTime
    paymentRecordMapper.updateById(pr);

    // 2. 批量更新关联账单: status→已缴费
    for (PaymentRecordBill d : details) {
        bill.setStatus(BillStatusEnum.PAID.getCode());
        billMapper.updateById(bill);
    }
}
// 任何一步抛异常 → 全部回滚
```

**原理**: MyBatis-Plus复用Spring事务管理器，同一事务内使用同一个数据库连接，利用MySQL的InnoDB事务机制保证原子性。

**边界情况处理**:
- 核销时支付记录已被其他人核销 → `if (!PENDING.equals(pr.getStatus())) throw ...` 乐观锁校验
- 数据库连接断开 → `@Transactional` 自动回滚
- @Transactional 默认只回滚 RuntimeException → 显式指定 `rollbackFor = Exception.class` 覆盖

### Q12: 项目中的 AI 功能是真的调用大模型吗？设计的巧妙之处在哪？

### Q13: 资产空间树是怎么构建的？时间复杂度是多少？

**答**:

```java
// AssetServiceImpl.java:32-53
public List<AssetTreeVO> getAssetTree() {
    List<House> houses = assetMapper.selectAllAssets();  // 1次查询
    Map<Long, List<House>> parentMap = houses.stream()
        .collect(Collectors.groupingBy(House::getParentId));  // O(n)
    return buildAssetTree(0L, parentMap);  // 递归, O(n)
}

private List<AssetTreeVO> buildAssetTree(Long parentId,
        Map<Long, List<House>> parentMap) {
    List<House> children = parentMap.get(parentId);  // O(1) HashMap查找
    for (House house : children) {
        treeVO.setChildren(buildAssetTree(house.getId(), parentMap));  // 递归
    }
    return treeVOs;
}
```

**复杂度分析**:
- 时间复杂度: **O(n)** — 每个节点访问一次
- 空间复杂度: **O(n)** — parentMap存储所有节点
- 数据库查询: **1次** — 避免N+1问题

**优势**: 一次查询，内存中构建树，适合中小规模数据
**局限**: 资产10万+时，数据库查询+内存压力大
**优化方向**: Redis缓存 + 资产变更时失效缓存

### Q14: 为什么购买/租赁审批通过后自动生成账单？失败了怎么办？

**答**: 这是**业务流程自动化**的体现——审批通过=合同生效=费用立即产生。

```java
// generateBillsSafe() — 安全的账单生成（失败不阻塞主流程）
public void generateBillsSafe(BillGenerateDTO dto) {
    try {
        doGenerateBills(dto);
    } catch (Exception e) {
        log.error("自动生成账单失败: ...", e);  // 记录日志
        // 不抛异常！不阻塞审批！
    }
}
```

**体现了最终一致性思想**:
- 审批是主流程 → 必须成功
- 账单生成是辅助流程 → 尽量成功，失败后可手动补救
- 日志记录确保管理员可以发现并处理失败的账单

### Q15: 项目的安全措施有哪些？能防范哪些攻击？

**答**:

**已实现的安全措施**:
| 措施 | 防御的攻击 |
|------|----------|
| BCrypt 密码加密 | 数据库泄露后密码不可逆 |
| JWT HMAC-SHA 签名 | Token篡改检测 |
| Token 24h过期 | Token长期滥用 |
| sessionStorage存Token | 比localStorage更短的生命周期 |
| CORS限制Header | 跨域请求伪造 |
| ThreadLocal清理 | 用户数据串扰 |
| @Valid参数校验 | 恶意输入 |
| MyBatis-Plus参数化查询 | SQL注入 |
| SecurityUtil.checkOwnership() | 越权访问 |

**当前缺失的安全措施** (可改进):
- **无Rate Limiting** → 可暴力破解登录接口（加 Guava RateLimiter 或 Nginx limit_req）
- **无Token黑名单** → 无法主动失效泄露的Token（加Redis黑名单）
- **无密码复杂度要求** → 弱密码风险（加 @Pattern 校验）
- **无HTTPS强制** → 中间人攻击（Nginx配置强制HTTPS）
- **错误信息区分** → 可枚举用户名（"USER_NOT_FOUND" vs "PASSWORD_ERROR"，改为统一的"用户名或密码错误"）

### Q16: 物业费单价"下月生效"是怎么实现的？

**答**:

```java
// 设置: 生效月 = 下个月
public void setPropertyFeeConfig(PropertyFeeConfigDTO dto) {
    String effectiveMonth = LocalDate.now().plusMonths(1)
        .format(DateTimeFormatter.ofPattern("yyyy-MM"));
    config.setEffectiveMonth(effectiveMonth);
    propertyFeeConfigMapper.insert(config);
}

// 查询(账单生成时): 匹配当月生效的配置
PropertyFeeConfig config = propertyFeeConfigMapper.getEffectiveConfig(billMonth);
// SQL: SELECT * FROM bus_property_fee_config
//      WHERE effective_month <= #{billMonth} AND status = 1
//      ORDER BY effective_month DESC LIMIT 1
```

**好处**: 本月设置的单价不影响本月已生成的账单，从下月开始按新标准。避免"月中调价"导致的财务纠纷。

### Q17: 如果登录接口被频繁调用（暴力破解），怎么保护？

**答**: 当前项目没有限流措施。改进方案（由轻到重）:

1. **Nginx 层**: `limit_req_zone $binary_remote_addr zone=login:10m rate=5r/m;` 限制单IP每分钟5次登录
2. **Spring层**: 使用 Guava `RateLimiter` 或 Resilience4j `@RateLimiter`
3. **应用层**: 登录失败N次后要求验证码（对接图形验证码服务）
4. **Redis**: `INCR login_attempts:{ip}` + `EXPIRE 60` → 滑动窗口计数，超阈值拒绝
5. **账户锁定**: 同一账户连续失败5次 → 锁定15分钟（Redis存储锁定状态）

### Q18: 为什么用 sessionStorage 而不是 Cookie 存 Token？

**答**:

| 对比维度 | sessionStorage | Cookie |
|---------|---------------|--------|
| **CSRF风险** | 无（JS显式读取，不自动发送） | 有（每次请求自动携带） |
| **生命周期** | 关闭标签页即清除 | 可持久化，取决于max-age |
| **跨标签页** | 不共享（本项目用CustomEvent做跨组件通信） | 共享 |
| **容量** | 5MB+ | 4KB |
| **XSS风险** | 有（JS可读取） | 有（JS可读取，httpOnly可缓解） |

**本项目选择sessionStorage的理由**:
- 物业管理后台 → 用户每次打开通常从登录开始
- 不需要"记住我"功能（Cookie的持久化优势用不上）
- CSRF天然免疫，无需额外配置CSRF Token
