## 一、项目概览

### 1.1 项目定义

智慧物业管理系统 (WisdomPM) 是一个面向小区/写字楼等物业管理场景的全栈 Web 应用，覆盖资产登记、合同租赁、收费缴费、报修工单、投诉建议、公告通知、数据看板等核心物业管理功能。

### 4.2 Result.java — 统一响应封装

```java
@Data
public class Result<T> implements Serializable {
    private Integer code;    // 成功=200, 失败固定=500 (注意:与HTTP状态码不同)
    private String msg;      // 成功="success", 失败=错误描述
    private T data;          // 响应数据

    public static <T> Result<T> success(T data) {
        // code=200, msg="success"
    }

    public static <T> Result<T> error(String msg) {
        // code=500, msg=msg, data=null
    }
}
```

> ⚠️ **注意**: `Result.error()` 始终设置 `code=500`，即使 HTTP 状态码为 4xx。前端应读取 HTTP Status Code 而非 body.code。

### 4.3 PageResult.java — 分页响应

```java
@Data
public class PageResult<T> implements Serializable {
    private long total;           // 总记录数
    private List<T> records;      // 当前页数据

    public PageResult(long total, List<T> records) { ... }
}
```

### 4.4 BaseContext.java — ThreadLocal 上下文

```java
public class BaseContext {
    public static ThreadLocal<Long> threadLocal = new ThreadLocal<>();

    public static void setCurrentId(Long id) { threadLocal.set(id); }
    public static Long getCurrentId() { return threadLocal.get(); }
    public static void removeCurrentId() { threadLocal.remove(); }
}
```

**用途**: 在 JWT Filter 中设置当前用户 ID，Service 层随时获取，无需通过方法参数传递。

> ⚠️ **安全性**: `threadLocal` 字段为 `public`，建议改为 `private`。

## 五、wisdom-pojo 模块详解

### 5.1 模块说明

wisdom-pojo 只包含实体类 Entity，与数据库表一一对应。**没有 DTO/VO 子包**——DTO 和 VO 全部放在 wisdom-server 中。

所有实体类统一规范：
- 使用 Lombok `@Data` 注解
- 实现 `java.io.Serializable`
- `@TableName` 标注对应数据库表名
- `@TableId(type = IdType.AUTO)` 标注自增主键
- 表名规则: `sys_*` 系统表 / `bus_*` 业务表

### 6.3 Filter 过滤层 (2个)

#### JwtAuthenticationFilter (@Order 1)

- **定位**: 被动信息提取，**永不拦截请求**
- **职责**: 
  1. 提取 `Authorization: Bearer <token>`
  2. 验证 JWT 有效性
  3. 解析 userId 存入 `BaseContext`
  4. `finally` 清理 ThreadLocal

#### AuthenticationFilter (@Order 2)

- **定位**: 主动鉴权门禁
- **白名单路径**:
  - `/api/v1/auth/login`
  - `/api/v1/auth/register`
  - `/swagger-ui/**`, `/api-docs/**`, `/v3/api-docs/**`, `/knife4j/**`
- **鉴权逻辑**:
  1. 白名单路径 → 直接放行
  2. 非白名单 → 通过 `RequestMappingHandlerMapping` 查找目标 Handler 方法
  3. 方法有 `@LoginRequired` → 检查 Token → 无效则 401
  4. 方法无 `@LoginRequired` → 公开接口，直接放行

### 6.4 Annotation 注解层 (2个)

|------|------|----------|
| `@LoginRequired` | 标记需要登录的接口 | 22个接口使用 |
| `@Anonymous` | 标记公开接口 | **已定义但未使用**（Filter 用白名单而非此注解） |

## 八、认证与鉴权体系

### 8.1 整体流程

```
客户端请求
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│ JwtAuthenticationFilter (@Order 1)                       │
│                                                          │
│  提取 Authorization 头: Bearer <token>                   │
│       │                                                  │
│       ├── 有token → JwtTokenUtil.validateToken()         │
│       │       ├── 有效 → BaseContext.setCurrentId(userId)│
│       │       └── 无效 → 忽略，不阻塞                     │
│       └── 无token → 继续                                 │
│                                                          │
│  finally: BaseContext.removeCurrentId()                  │
└─────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│ AuthenticationFilter (@Order 2)                          │
│                                                          │
│  1. 检查白名单匹配?                                       │
│     ├── 是 (/api/v1/auth/login, /register, /swagger等)   │
│     │   → 直接放行                                       │
│     └── 否 → 进入步骤2                                   │
│                                                          │
│  2. 通过 RequestMappingHandlerMapping 查找 Handler 方法   │
│     ├── @LoginRequired 注解存在?                          │
│     │   ├── 是 → 3. 检查Token                            │
│     │   │       ├── 有效 → BaseContext.setCurrentId → 放行│
│     │   │       └── 无效/缺失 → 返回 401 JSON             │
│     │   └── 否 → 公开接口，放行                           │
│                                                          │
│  finally: BaseContext.removeCurrentId()                  │
└─────────────────────────────────────────────────────────┘
```

### 8.2 鉴权特点总结

| 特点 | 说明 |
|------|------|
| 无 Spring Security | 不依赖 Spring Security 框架，仅用其 BCrypt |
| 基于注解 | `@LoginRequired` 标记接口需要鉴权 |
| 基于白名单 | 登录/注册/Swagger 路径直接放行 |
| 默认公开 | 无 `@LoginRequired` 的接口 = 公开接口 |
| 无状态 | JWT，服务端不存储 Session |
| ThreadLocal | 用户ID通过 BaseContext 传递，不污染方法签名 |
| 无主动登出 | Token 签发后无法撤销（未实现黑名单） |
| 无 Token 刷新 | 过期后需重新登录 |

## 九、异常处理机制

### 9.1 异常类设计

```
RuntimeException
    └── BusinessException (code + message)
            ├── unauthorized()    → code=401
            ├── forbidden()      → code=403
            ├── notFound(msg)    → code=404
            └── badRequest(msg)  → code=400
```

### 9.2 处理流程

```
Service 层:
  throw BusinessException.notFound("房产不存在")
  throw BusinessException.badRequest("仅待缴费状态的账单可支付")
  throw BusinessException.forbidden("无权限操作")

     ↓

GlobalExceptionHandler (@RestControllerAdvice):
  @ExceptionHandler(BusinessException.class)
  {
      switch (e.getCode()):
          401 → ResponseEntity.status(401).body(Result.error(e.getMessage()))
          403 → ResponseEntity.status(403).body(Result.error(e.getMessage()))
          404 → ResponseEntity.status(404).body(Result.error(e.getMessage()))
          400 → ResponseEntity.status(400).body(Result.error(e.getMessage()))
          default → ResponseEntity.status(200).body(Result.error(e.getMessage()))
  }

     ↓

前端接收:
  HTTP/1.1 404 Not Found
  Content-Type: application/json
  { "code": 500, "msg": "房产不存在", "data": null }
```

> ⚠️ **已知问题**: `Result.error()` 固定设置 body.code=500，但 HTTP Status 正确。前端应读取 HTTP Status 而非 body.code。

```java
private BigDecimal calculateBillAmount(House house, Contract contract, Integer feeType) {
    return switch (feeType) {
        case RENT, PURCHASE → contract.getRentAmount();
        case DEPOSIT → contract.getDeposit();
        case PROPERTY → house.getArea().multiply(unitPrice).setScale(2, HALF_UP);
    };
}
```

## 十五、核心业务流程

### 15.1 缴费系统完整流程 ⭐ (最复杂)

```
┌─────────────────────────────────────────────────────┐
│  第一步：生成账单 (POST /api/v1/bills/generate)      │
│                                                      │
│  输入: feeType, billMonth, houseId, remark           │
│                                                      │
│  → 确定账期 (物业费=下月, 其他=当月/指定月)            │
│  → 属性费需查 PropertyFeeConfig (最近有效配置)         │
│  → 遍历目标房产                                       │
│      ├── 跳过 BUILDING 类型                           │
│      ├── 跳过无业主 (ownerId==null)                    │
│      ├── 去重检查 (houseId+feeType+billMonth 唯一)     │
│      ├── 计算金额:                                     │
│      │     RENT/PURCHASE → contract.rentAmount         │
│      │     DEPOSIT → contract.deposit                  │
│      │     PROPERTY → house.area × config.unitPrice    │
│      ├── 设置 dueDate = 账期月末                        │
│      └── 插入 bus_bill (status=PENDING)               │
│  → 生成张数=0? → throw BusinessException              │
│                                                      │
│  输出: 生成的账单列表 (status=PENDING)                 │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│  第二步：用户缴费 (POST /api/v1/payments)             │
│                                                      │
│  输入: billIds[], payMethod, voucherUrl, remark      │
│                                                      │
│  → 校验: 所有账单存在 + 属于当前用户 + 状态=PENDING     │
│  → 计算 totalExpected = sum(bill.amount)             │
│  → 创建 PaymentRecord (status=PENDING)                │
│  → 创建 PaymentRecordBill[] (每条对应一个billId)      │
│  → 更新 Bill[].status = PENDING → SUBMITTED          │
│                                                      │
│  输出: 缴费记录 (status=PENDING, "待核销")            │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│  第三步：管理员核销 (PUT /api/v1/payments/{id}/verify)│
│                                                      │
│  → 校验: PaymentRecord 存在 + 状态=PENDING            │
│  → PaymentRecord → VERIFIED + verifyUser + verifyTime│
│  → 关联 Bill[] → PAID                                │
│                                                      │
│  输出: 缴费记录 status=VERIFIED, 账单 status=PAID    │
└─────────────────────────────────────────────────────┘
                         │
                         ▼ (可选撤销)
┌─────────────────────────────────────────────────────┐
│  第四步：撤销核销 (PUT /api/v1/payments/{id}/cancel)  │
│                                                      │
│  输入: cancelReason                                  │
│                                                      │
│  → 校验: PaymentRecord 存在 + 状态=VERIFIED           │
│  → PaymentRecord → CANCELLED + cancelUser+Time+Reason│
│  → 关联 Bill[] → PAID 回退为 PENDING                  │
│                                                      │
│  输出: 缴费记录 status=CANCELLED, 账单 status=PENDING │
└─────────────────────────────────────────────────────┘
```

### 15.2 账单状态机

```
        ┌──────────┐
        │  PENDING │  ← 初始生成 / 核销撤销后回退
        │  待缴费   │
        └─────┬────┘
              │ submitPayment()
              ▼
        ┌──────────┐
        │SUBMITTED │  ← 用户已提交缴费，等待管理员确认
        │ 待核销    │
        └─────┬────┘
              │ verifyPayment()
              ▼
        ┌──────────┐
        │   PAID   │  ← 管理员确认到账
        │  已缴费   │
        └─────┬────┘
              │ cancelPayment()
              ▼
        ┌──────────┐
```
