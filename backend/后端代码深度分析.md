## 一、认证鉴权体系（源码级剖析）

### 1.1 双 Filter 架构设计

```
┌─────────────────────────────────────────────────────┐
│ Filter 链执行顺序 (@Order 控制)                       │
│                                                      │
│  Step 1: JwtAuthenticationFilter (@Order=1)          │
│          → 被动模式：提取Token但不拦截请求              │
│          → 成功解析后存入 BaseContext ThreadLocal      │
│          → 失败则静默跳过，交由下一个Filter处理         │
│                                                      │
│  Step 2: AuthenticationFilter (@Order=2)             │
│          → 主动模式：白名单放行 + @LoginRequired 鉴权  │
│          → 白名单命中 → chain.doFilter 直接放行        │
│          → 非白名单 → 反射检查 HandlerMethod 注解       │
│          → @LoginRequired + 无Token → 返回401 JSON    │
│          → @LoginRequired + Token有效 → 放行          │
│          → 无@LoginRequired → 放行（默认公开）         │
└─────────────────────────────────────────────────────┘
```

### 1.2 关键源码解读

**JwtAuthenticationFilter 的设计意图**（源码第28-36行）:

```java
// 设计要点: 这是一个"只提取、不拦截"的过滤器
// 即使Token无效，也不会返回401，而是交给AuthenticationFilter处理
if (token != null && token.startsWith("Bearer ")) {
    token = token.substring(7);
    if (jwtTokenUtil.validateToken(token)) {
        Long userId = jwtTokenUtil.getUserIdFromToken(token);
        if (userId != null) {
            BaseContext.setCurrentId(userId);  // ← 存入ThreadLocal
        }
    }
}
// 关键: 不拦截！链继续执行
try {
    chain.doFilter(request, response);
} finally {
    BaseContext.removeCurrentId();  // ← 防止内存泄漏
}
```

**设计评价**: 这种"双Filter分离职责"的设计值得肯定——JWT解析和鉴权决策解耦，各自职责单一。但存在一个小问题：两个Filter都调用了 `chain.doFilter`，意味着即使 JwtAuthenticationFilter 已经成功设置了 BaseContext，AuthenticationFilter 仍然会重新解析Token一次，造成重复解析。

**AuthenticationFilter 的鉴权逻辑**（源码第109-124行）:

```java
// 核心: 通过 Spring RequestMappingHandlerMapping 反射获取HandlerMethod
// 然后检查方法上的 @LoginRequired 注解
private boolean requiresLogin(HttpServletRequest request) {
    HandlerExecutionChain handlerChain = handlerMapping.getHandler(request);
    if (handlerChain == null) return false;  // 静态资源 → 不需要登录
    if (handlerChain.getHandler() instanceof HandlerMethod handlerMethod) {
        Method method = handlerMethod.getMethod();
        return method.isAnnotationPresent(LoginRequired.class);  // ← 注解驱动
    }
    return false;
}
```

**设计评价**: 
- ✅ 优点：利用了Spring MVC的HandlerMapping机制，不需要维护URL配置
- ⚠️ 风险：`handlerMapping.getHandler(request)` 会触发Spring MVC的完整Handler查找链，包括拦截器匹配，在高并发下有一定性能开销
- ⚠️ 隐患：如果HandlerMapping查找失败（异常），返回false意味着"不需要登录"——这是一种"fail-open"策略，安全性偏向宽松

### 1.3 BaseContext ThreadLocal 分析

```java
// wisdom-common/src/main/java/com/wisdom/context/BaseContext.java
public class BaseContext {
    public static ThreadLocal<Long> threadLocal = new ThreadLocal<>();  // ⚠️ public!
}
```

**两个问题**:
1. **`threadLocal` 字段是 `public` 的**: 任何代码都可以直接操作 `BaseContext.threadLocal.set(xxx)`，破坏封装性
2. **基于Tomcat线程池的ThreadLocal是安全的**: 因为两个Filter都在 finally 块中调用了 `removeCurrentId()`，保证了线程归还线程池前清理了ThreadLocal

```java
    @Value("${jwt.expiration:86400000}") long expiration  // 默认24小时
```


### 2.3 合同服务与资产的联动（ContractServiceImpl.saveOrUpdateContract）

**新增合同的完整流程**:

```java
@Transactional(rollbackFor = Exception.class)  // ← 事务边界
public void saveOrUpdateContract(ContractDTO contractDTO) {
    if (contractDTO.getId() == null) {
        // 1. 插入合同
        contractMapper.insert(contract);
        
        // 2. 更新房产状态
        if (house != null && contractStatus == 1) {
            house.setStatus("RENTING");
            house.setOwnerName(contract.getTenantName());
            assetMapper.updateById(house);
            
            // 3. 自动生成账单（非阻塞模式）
            // 注意: 每个账单生成独立 try-catch，互不影响
            try {
                financeService.generateBillsSafe(rentDTO);   // 租金
            } catch (Exception e) { /* 不阻塞 */ }
            try {
                financeService.generateBillsSafe(depositDTO); // 押金
            } catch (Exception e) { /* 不阻塞 */ }
            try {
                financeService.generateBillsSafe(propertyDTO); // 物业费
            } catch (Exception e) { /* 不阻塞 */ }
        }
    } else {
        // 终止合同: 合同状态→0, 房产→VACANT, 清空业主
        contractMapper.updateById(contract);
        if (contractStatus == 0) {
            house.setStatus("VACANT");
            house.setOwnerName(null);   // ← 清空业主
            house.setOwnerPhone(null);
            house.setOwnerId(null);     // ← ⚠️ 清空业主ID!
            assetMapper.updateById(house);
        }
    }
}
```

**设计评价**:
- ✅ 事务边界合理（合同 + 房产状态必须在同一事务）
- ✅ 账单生成容错设计（非阻塞 try-catch）
- ⚠️ **终止合同时直接清空 `ownerId`**——这意味着房屋的业主信息被完全清除，历史数据无法追溯该房产曾经属于谁
- ⚠️ `generateBillsSafe` 方法内部也有 `@Transactional`，而外层 `saveOrUpdateContract` 也有事务——嵌套事务需要注意 Spring 的传播行为（默认 REQUIRED 会加入外层事务）

### 3.1 异常处理链

```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    // 优先级最高: 业务异常
    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<Result<Void>> businessExceptionHandler(BusinessException ex) {
        HttpStatus status = switch (ex.getCode()) {
            case 401 -> HttpStatus.UNAUTHORIZED;
            case 403 -> HttpStatus.FORBIDDEN;
            case 404 -> HttpStatus.NOT_FOUND;
            case 400 -> HttpStatus.BAD_REQUEST;
            default  -> HttpStatus.OK;           // ← 其他code返回200!
        };
        return ResponseEntity.status(status).body(Result.error(ex.getMessage()));
    }
    
    // 参数校验异常
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Result<Void>> validationExceptionHandler(...) {
        return ResponseEntity.badRequest().body(Result.error(message));
    }
    
    // 兜底: RuntimeException → 500
    @ExceptionHandler(RuntimeException.class)
    public ResponseEntity<Result<Void>> runtimeExceptionHandler(...) {
        return ResponseEntity.status(500).body(Result.error("服务器内部错误"));
    }
    
    // 最终兜底: Exception → 500
    @ExceptionHandler(Exception.class)
    public ResponseEntity<Result<Void>> exceptionHandler(...) {
        return ResponseEntity.status(500).body(Result.error("服务器内部错误"));
    }
}
```

### 3.2 Result.body.code 与 HTTP Status 不一致问题

**问题演示**:

```
客户端请求: DELETE /api/v1/assets/99999 (不存在的资产)
  ↓
Service 抛出: BusinessException.notFound("资产不存在")  // code=404
  ↓
GlobalExceptionHandler 处理:
  HTTP Status: 404  ← 正确
  Response Body: { "code": 500, "msg": "资产不存在", "data": null }
                  ↑ code始终是500!
```

**根源**: `Result.error()` 方法硬编码了 `code = 500`:

```java
public static <T> Result<T> error(String msg) {
    Result<T> result = new Result<>();
    result.msg = msg;
    result.code = 500;  // ← 永远500，不管实际是什么错误
    return result;
}
```

**影响**: 如果前端只读 `body.code` 而不读 HTTP Status（这在一些项目中很常见），会认为所有错误都是 500 内部错误。

**修复方案**:
```java
// 新增支持自定义code的工厂方法
public static <T> Result<T> error(int code, String msg) {
    Result<T> result = new Result<>();
    result.code = code;
    result.msg = msg;
    return result;
}
```

## 四、数据访问层分析

### 4.1 MyBatis-Plus 用法评估

**使用模式**: LambdaQueryWrapper + Page + BaseMapper

```java
// 标准的分页查询模式（在项目中大量使用）
Page<Bill> page = new Page<>(pageNum, pageSize);
LambdaQueryWrapper<Bill> qw = new LambdaQueryWrapper<>();
qw.eq(Bill::getStatus, status)          // 类型安全的条件构建
  .orderByDesc(Bill::getCreateTime);
IPage<Bill> result = billMapper.selectPage(page, qw);
```

**评价**:
- ✅ 使用 Lambda 表达式构建查询条件，类型安全，避免字符串硬编码
- ✅ 分页查询统一使用 MyBatis-Plus 的 Page + IPage 模式
- ✅ 动态条件: 先判空再添加条件
- ⚠️ 权限过滤条件分散在各个 Service 中，重复代码较多

## 六、架构设计决策分析

### 6.1 为什么不用 Spring Security？

**实际实现**: 两个自定义 Servlet Filter + @LoginRequired 注解

**合理之处**:
- 系统只有"管理员"和"普通用户"两种角色，权限模型简单
- Spring Security 引入的配置复杂度 > 自实现的复杂度
- Filter + 注解的方式代码量不到 200 行，非常轻量

**不足之处**:
- 缺少防暴力破解（登录失败次数限制）
- 无法集成 OAuth2/OIDC
- 缺少方法级权限控制（如 `@PreAuthorize`）

### 6.2 为什么 DTO/VO 不放在 wisdom-pojo？

**现状**: 所有 DTO (27个) 和 VO (18个) 都在 wisdom-server 模块

**实际原因**: 开发便利性——DTO/VO 和 Controller/Service 紧密相关，放在同一模块减少了跨模块引用

**建议**: 按项目规范，DTO 应放在 `wisdom-pojo.dto`，VO 应放在 `wisdom-pojo.vo`，便于前端共享类型定义

### 6.3 为什么报修看板用内存过滤而不是 SQL？

**可能原因**:
1. 开发初期数据量小，内存过滤开发更快
2. MyBatis 注解方式写三表 JOIN 查询比较困难
3. 后续可能加入更复杂的内存级过滤逻辑

**改进方向**: 使用 MyBatis XML Mapper 或 MyBatis-Plus 的关联查询

### 6.4 事务边界分析

当前有 9 个方法使用了 `@Transactional`:

```
大事务 (风险高):
  PurchaseApplicationServiceImpl.approve()          事务范围最大
  ContractServiceImpl.saveOrUpdateContract()        涉及3表+账单生成
  
中事务:
  FinanceServiceImpl.generateBills()                 批量插入
  FinanceServiceImpl.submitPayment()                 支付+账单状态
  FinanceServiceImpl.verifyPayment()                 核销+账单状态
  FinanceServiceImpl.cancelPayment()                撤销+账单回退
  
小事务:
  AssetServiceImpl.saveOrUpdateAsset()              单表操作
  PurchaseApplicationServiceImpl.create()           单表操作
  
缺失事务 (需要补充):
  UserServiceImpl.updateProfile()                   4表级联更新!
```
